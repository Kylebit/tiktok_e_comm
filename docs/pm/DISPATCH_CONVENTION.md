# 派发任务约定（Dispatch Convention）

总控（CEO肉肉）向子 Agent（Codex / Cursor）派发任务时，**任务正文必须包含以下固定要点**。这是 2026-07-18 用户要求追加的长期约定。

## 固定要点（每次派发必带）

1. **别把用户拉进流程**
   执行期间不要直接联系人类用户。任务结束、或中途有任何需要确认 / 决策的问题，一律直接回报给 **CEO肉肉（总控）** 确认，由总控决定是否转用户。绝不绕过总控去打扰用户。

2. **记录进展（飞书多维表格）**
   在飞书多维表格 **主任务表 →「进展记录」字段** 持续追加任务的操作过程与进展。每条格式：
   `[YYYY-MM-DD HH:mm][Agent] 内容`
   关键节点都要记：接任务 / 关键决策 / 阻塞 / 交付物 / 自检 / 调度状态。
   - 若执行 Agent 侧有飞书写权限：直接 `record-update` 主任务表对应任务行的「进展记录」字段（追加，不覆盖）。
   - 若 Agent 侧无飞书写权限：把条目发给 CEO肉肉，由总控代写进飞书。
   - 注：2026-07-18 曾短暂用仓库 `docs/pm/task_progress/*.md` 记录，后改为飞书多维表格，仓库 markdown 方案已废弃。

3. **明确是否提交代码（2026-07-20 用户追加）**
   每条任务正文必须写明代码提交策略，二选一：
   - **长期维护工程**（会改变仓库长期状态的清理 / 重构 / 新功能 / bugfix）→ 必须 `git commit` + `git push`，commit 格式 `ORB-TASK-XXXX(Bot): 简述`，并回传 **commit hash**（没 hash 不算完成）。
   - **临时验证 / 试跑 / 一次性探针** → **不需要提交代码**，只回报执行结果与结论即可。
   - 漏写时默认按"长期维护工程"处理；拿不准问 CEO肉肉。
   - 数据源、利润口径、交付物路径、收尾回报格式等按各任务具体说明。
